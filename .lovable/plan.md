

## Deep Audit & Fix: Invoice Line-Item Pricing Consistency + Asset-Wise Billing

### Root Cause Analysis

**Problem 1: Invoice preview shows wrong line amounts**

INV/2026-27/0010 has JSONB items storing correct values (`rent_amount: 70000`, `rate: 70000`, `amount: 70000`). However, the enrichment in `InvoiceTemplateZoho.tsx` (line 172) checks `item.rent_amount != null` — which works. But the **real issue** is the JSONB items generated by `MonthlyInvoiceGenerator.tsx` (line 515-534) store items with `rate` and `amount` fields but **omit critical fields**: `rent_amount`, `campaign_asset_id`, `booking_start_date`, `booking_end_date`, `booked_days`, `dimensions`, `total_sqft`, `illumination_type`.

Because `campaign_asset_id` is missing, the enrichment at line 167 (`const ca = item.campaign_asset_id ? caMap.get(...)`) returns `undefined`, so the template falls back to fetching metadata from `media_assets`. This means for **this specific invoice**, the enrichment doesn't pull corrupt `campaign_assets.rent_amount`. The ₹1,75,664 the user sees may come from a stale browser cache or a different rendering path.

However, **many other invoices** (confirmed by database audit) have JSONB items where `rent_amount` stores the **total campaign-level rate** (not the prorated monthly amount), causing a systemic mismatch between displayed line totals and invoice `sub_total`.

**Problem 2: Campaign billing uses header dates, not asset-wise dates**

The `calculateBillingPeriods()` in `computeCampaignTotals.ts` uses campaign-level `campaignPeriodStart/End` (derived from min/max of asset dates). This works for period generation, but the `calculatePeriodAmountFromTotals()` distributes the total `displayCost` proportionally across periods rather than computing per-asset overlap per period. For mixed-start-date campaigns, this can produce misleading per-month amounts.

The `MonthlyInvoiceGenerator` already does asset-wise billing correctly (line 160-238: `calculateAssetBilling` computes per-asset overlap with the billing month). The problem is the **billing schedule table** (`MonthlyBillingScheduleTable`) uses the campaign-level distribution from `calculatePeriodAmountFromTotals`, which doesn't match what the invoice generator actually bills.

#### Fix 2: Harden template enrichment to never override stored pricing

**File: `src/components/invoices/InvoiceTemplateZoho.tsx`** (lines 164-198)

The current enrichment already prioritizes invoice JSONB values (line 172). But strengthen with explicit comments and a guard:

```typescript
// CRITICAL: Invoice JSONB items are the IMMUTABLE source of truth for pricing.
// campaign_assets/media_assets are only used to backfill MISSING metadata
// (location, dimensions, etc.), NEVER to override financial values.
const rentAmount = item.rent_amount ?? item.rate ?? item.amount ?? (ca?.rent_amount ?? 0);
```

Wait — the current code at line 172 is:
```typescript
const rentAmount = item.rent_amount != null ? item.rent_amount : (ca?.rent_amount ?? item.rent_amount);
```

This is correct when `item.rent_amount` exists. But for older invoices where JSONB has `rent_amount` but it stores the **campaign total** (not monthly), this will display wrong values. The fix must also ensure `item.rate` and `item.amount` are checked — these are the fields actually stored by MonthlyInvoiceGenerator.

Updated enrichment logic:
```typescript
// Pricing: prefer stored billed amount, not campaign-level totals
const storedPrice = item.rent_amount ?? item.rate ?? item.amount;
const rentAmount = storedPrice != null ? storedPrice : (ca?.rent_amount ?? 0);
const printingCharges = item.printing_charges != null ? item.printing_charges : (ca?.printing_charges ?? 0);
const mountingCharges = item.mounting_charges != null ? item.mounting_charges : (ca?.mounting_charges ?? 0);
// Recalculate line total from components — never trust pre-computed totals from external sources
const lineTotal = (rentAmount || 0) + (printingCharges || 0) + (mountingCharges || 0);
```

Also add the same guard at line 311 (render section):
```typescript
const rentAmount = item.rent_amount || item.rate || item.amount || 0;
```
This line is already correct. But change line 314 to always recalculate:
```typescript
const lineTotal = (rentAmount || 0) + (printingCharges || 0) + (mountingCharges || 0);
```
Remove the fallback to `item.total || item.amount`.

#### Fix 4: Align billing schedule table with actual asset-wise computation

**File: `src/components/campaigns/billing/MonthlyBillingScheduleTable.tsx`**

Currently uses `calculatePeriodAmountFromTotals()` which distributes campaign total proportionally. Instead, compute per-period amounts by summing asset-wise billing (same logic as `MonthlyInvoiceGenerator.calculateAssetBilling`).

Add a new utility function in `computeCampaignTotals.ts`:
```typescript
export function calculatePeriodAmountAssetWise(
  period: BillingPeriodInfo,
  campaignAssets: CampaignAsset[],
  totals: CampaignTotalsResult,
  includePrinting: boolean,
  includeMounting: boolean,
): { baseRent: number; printing: number; mounting: number; subtotal: number; gstAmount: number; total: number }
```

This function iterates each active asset, computes overlap days with the period, calculates `(monthlyRate / 30) * overlapDays`, and sums. This matches what `MonthlyInvoiceGenerator` actually bills.

### Historical Invoice Audit

The database audit found **~40+ invoices** where JSONB item rent totals don't match `sub_total`. These fall into categories:

1. **Legacy invoices** (INV-202602-xxxx, older INV/2025-26/xxxx): Items have no pricing fields at all (sum = 0 or null). These rely entirely on enrichment.
2. **Monthly split invoices** (INV-Z/2025-26/xxxx series): Items store campaign-level `rent_amount` (total for full booking) instead of prorated monthly amount.
3. **Recent invoices**: Correctly stored after previous fixes.

**Recommendation**: Do NOT auto-fix paid invoices. Generate a flagged list for manual review. Fix the rendering so even old invoices display correctly by preferring `sub_total / item_count` as a sanity-check fallback.
